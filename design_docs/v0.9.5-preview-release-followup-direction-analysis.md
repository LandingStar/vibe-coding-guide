# v0.9.5 Preview Release Follow-up Direction Analysis

## Context

当前 `0.9.5` preview release 已完成以下闭环：

1. `Project Progress Global Direction-Candidates Artifact Consistency Audit` 已完成并关闭
2. `.codex/progress-graph/latest.json`、`.dot`、`.html` 已按最新状态重新生成
3. dual-package 版本面、release 文档与 `pack-lock.json` 已同步到 `0.9.5`
4. `scripts/release.py --no-isolation` 已跑通，产出 `release/doc-based-coding-v0.9.5.zip`

当前仓库已经回到无 active planning-gate 状态，但下一步方向还没有重新收口为新的窄文档入口。

## Candidate A. Companion Prose Projection Recovery

- 做什么：回到此前被 audit 暂停的主线，为 `用户选定下一步`、`当前更窄的入口`、`因此当前实际下一条 planning-gate` 这类 companion prose 提供最小语义投影
- 依据：`design_docs/direction-candidates-after-phase-35.md`、`design_docs/project-progress-global-direction-candidates-recency-semantics-followup-direction-analysis.md`
- 为什么现在适合做：本轮已经把真实 artifact consistency 风险和 stale artifact 风险清掉，继续回到 companion prose projection，能够直接提升 progress graph 对“为什么当前走到这一步”的可解释性
- 风险：中。会继续触碰 `tools/progress_graph/doc_projection.py` 的 source-selection contract，但属于已知邻接面
- 当前判断：推荐。它直接承接用户先前明确选过、但被 audit 中断的方向，而且收益最贴近刚完成的 progress graph 主线

## Candidate B. Release-Close Handoff / CURRENT Refresh Hardening

- 做什么：围绕本次 release-close 过程中暴露出的状态面漂移，收一条窄 slice，把 release safe stop 与 `.codex/handoffs/CURRENT.md` / authority-doc footprint 再次对齐
- 依据：`design_docs/Project Master Checklist.md` 当前 `Current Handoff Footprint` 仍停留在旧 handoff；`design_docs/tooling/Document-Driven Workflow Standard.md` 要求验证后回写 handoff；`review/research-compass.md` 中 docs/template/platform 与 agent lifecycle 研究都强调稳定控制面与可恢复入口
- 为什么现在适合做：这次 `0.9.5` release 已经证明 release 本体可产出，但 safe-stop / CURRENT 的更新仍靠人工串联，属于已发现且未彻底机制化的后续 gap
- 风险：中。文档/流程层收益明确，但用户可见能力增量小于 Candidate A
- 当前判断：次优先。若你更在意 release-close 的恢复一致性，而不是 graph 解释能力，可以先做这条

## Candidate C. Post-Release Dogfood / Install Path Tightening

- 做什么：以 `release/doc-based-coding-v0.9.5.zip` 为输入，继续做 clean-venv 安装、CLI smoke，或进一步收窄到 extension 从 release zip 安装 runtime 的体验入口
- 依据：`design_docs/Project Master Checklist.md` 中“持续 pre-release dogfood”待办；`docs/first-stable-release-boundary.md`；`design_docs/direction-candidates-after-phase-35.md` 中关于 extension 从 release zip 安装 Python runtime 的候选；`review/research-compass.md` 中关于 docs-like-code、templates、pack lifecycle 的研究入口
- 为什么现在适合做：当前已经有真实 `0.9.5` 产物，继续做受控 dogfood 会比只在 workspace 内跑测试更接近外部分发路径
- 风险：中。很容易从“验证安装”扩到“安装向导/UX”实现，需要严格维持窄 scope
- 当前判断：适合当你想把这次 release 从“已构建”进一步推进到“已按目标安装路径被验证”时进入

## Recommendation

我当前倾向于优先进入 Candidate A。

原因很直接：

1. 这条线是本轮 audit 前用户已经明确选过的主线，只是被 artifact consistency audit 临时打断
2. 现在 audit 与 release blocker 都已经清掉，再回到 companion prose projection 能延续 progress graph 主线，而不是切去新的运维/流程子题
3. Candidate B 和 C 都合理，但分别偏向恢复治理和分发验证；它们更像 release 之后的稳态强化，而不是当前最自然的主链延续

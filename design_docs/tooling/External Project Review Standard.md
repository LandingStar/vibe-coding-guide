# External Project Review Standard

## 文档定位

本文件定义分析外部项目/资源时的标准流程。目标是降低重复劳动、保证分析深度一致性、并将分析产出结构化为可消费的行动项。

适用场景：

- 分析竞品/同类项目的代码仓库
- 阅读技术博客并提取可借鉴模式
- 评审外部架构方案的适用性

产出存放位置：`review/` 目录。

## 标准流程（5 步）

### Step 1 — 快速概览

**输入**：外部资源 URL（仓库 / 博客 / 文档）

**动作**：

1. 获取 README.md / 首页内容
2. 扫描仓库结构（若为代码仓库）
3. 识别核心思路、角色划分、文档职责

**产出**：

- 来源信息（URL、发现日期）
- 一段话项目概述
- 核心角色/文档/机制的列表

### Step 2 — 结构化对比

**动作**：将外部项目的关键维度与本平台做 1:1 映射。

**必须覆盖的对比维度**：

| 维度 | 说明 |
|------|------|
| 角色划分 | 外部项目的角色 ↔ 本平台的概念映射 |
| 文档职责 | 外部项目的文档 ↔ 本平台的文档映射 |
| 执行机制 | 规则/提示词层 vs 运行时层 |
| 治理验证 | 是否有机器可审计的约束检查 |
| 防漂移机制 | 显式规则 vs 隐式约定 |
| 持续推进机制 | 自动继续 vs 等待确认 |
| 复杂度定位 | 轻量模板 vs 完整平台 |

**产出**：对比表格（Markdown 表）

### Step 3 — 借鉴点提取

**动作**：从对比结果中提取值得关注的模式，逐条标注可操作性。

**每个借鉴点必须包含**：

```markdown
#### [借鉴点名称]

- **外部实现**：<外部项目如何实现此机制>
- **本平台现状**：<我们是否已有类似能力，覆盖度如何>
- **差异**：<关键差异点>
- **可操作性**：✅ 可直接采纳 / ⚠ 需适配 / ❌ 不适用 / 📋 记录为 future direction
- **采纳建议**：<若可操作，具体如何引入>
```

### Step 4 — 差异分析

**动作**：双向分析——外部项目缺什么 + 本平台缺什么。

**产出**：

- 外部项目缺失清单（验证我们的定位差异是否有意义）
- 本平台缺失清单（识别可改进点）
- 交叉验证结论（两个项目的演进路径是否互相验证）

### Step 5 — 行动项生成

**动作**：将 Step 3 和 Step 4 的结论转化为结构化行动项。

**行动项决策表格式**：

```markdown
| # | 建议 | 决策 | 理由 | 优先级 |
|---|------|------|------|--------|
| A1 | ... | ✅ 采纳 / ❌ 不采纳 / 📋 记录 | ... | 高/中/低/未来 |
```

**决策分类**：

- ✅ 采纳：可直接执行，需标注执行方式（规则层 / 代码层 / 文档层）
- ❌ 不采纳：不适合当前架构/定位，需标注理由
- 📋 记录：有价值但超出当前阶段，记录为 future direction

## Review 文档模板

```markdown
# [项目/资源名称] 分析

## 来源

- 仓库/URL：[链接]
- 发现日期：YYYY-MM-DD
- 状态：**初步分析** / **详细分析完成**

## 项目概述

<Step 1 产出>

## 结构化对比

<Step 2 对比表格>

## 借鉴点

<Step 3 逐条借鉴点>

## 差异分析

<Step 4 双向缺失分析 + 交叉验证>

## 行动项决策汇总

<Step 5 决策表格>
```

## 质量门

一份完整的 review 文档必须满足：

1. 所有 5 个 Step 的产出均已填写
2. 借鉴点至少有 3 个标注了可操作性
3. 行动项决策表包含了所有"✅ 采纳"项的具体执行方式
4. "不采纳"项标注了理由
5. 状态标记为"详细分析完成"

## 与其他标准的关系

- 产出的 ✅ 采纳项若涉及代码变更，应按 `Document-Driven Workflow Standard` 先生成 planning-gate
- 产出的 📋 记录项应写入 `design_docs/Project Master Checklist.md` 风险/待办
- review 文档本身存放在 `review/` 目录，不进入 `design_docs/`

# 📄 子文档 3-C：全量状态快照的数据结构与覆盖流程

**(State Snapshot & World Overwrite Protocol)**

## 1. 架构定位：预测失败时的终极保底

在物理隔离的 C/S 架构中，客户端允许进行本地预测，但**服务端世界始终是唯一真理**。

当客户端预测失败、断线重连、或观战者半途加入时，单纯依赖事件流已不足以恢复当前宇宙状态。此时引擎必须提供一套**强制时空覆盖**能力：

- 服务端将当前世界打包为权威快照。
- 客户端销毁本地错误世界。
- 客户端利用快照重建世界，并在必要时静默快进到当前本地时刻。

本子文档是[[base event function and communication system|文档 3：通信协议与状态同步架构设计规范]]中 `3.1 ~ 3.3` 的正式工程落地补完。

---

## 2. 数据哲学：只传“重建世界所需的最小原材料”

快照的目标不是“传输一棵活着的对象树”，而是传输**重建一个正确世界所需的最小数据原料**。

因此，状态快照必须遵循以下原则：

- **只传数据，不传行为。**
    
    快照中不允许出现函数、闭包、对象引用、监听器实例或运行时回调。
    
- **只传身份，不传指针。**
    
    所有跨对象关联均使用 `Entity_ID`、`Buff_Instance_ID` 等稳定 ID 表达。
    
- **只传原料，不传冗余派生值。**
    
    能由客户端本地工厂与规则表重建的内容，不应直接随快照传输整个运行时对象。

这与[[Engine Macro-Architecture|文档 0：微内核与依赖注入]]的设计纲领一致：网络层传输的是冰冷数据，本地 `Core` 与标准组件负责复活这些数据并恢复行为。

## 3. 快照的标准数据结构

## 3.1 顶层信封 (Snapshot Envelope)

一份 `GameWorldSnapshot` 至少应包含：

- `snapshot_id`
    
    快照自身的唯一标识，用于去重与调试追踪。
    
- `authoritative_tick`
    
    本快照所对应的服务端权威时间刻。
    
- `world_revision`
    
    世界状态版本号。客户端可据此拒收旧快照。
    
- `last_accepted_command_id` (可选但强烈推荐)
    
    服务端已确认接纳到哪条本地指令。用于客户端在覆盖后清理重播缓冲。
    
- `driver_type`
    
    当前行动权驱动类型，例如 `TIMELINE_AV` 或 `STRICT_TURN`。
    
- `global_payload`
    
    世界级全局状态，如阶段、天气、场景变量、回合计数等。

> **[推断补完说明]**
> 
> 历史对话已明确提出“强快照同步”和“静默快进”的需求，但未完全定型顶层字段。为使该机制可实施，此处将 `authoritative_tick`、`world_revision` 与 `last_accepted_command_id` 作为推荐基线字段。

## 3.2 驱动器状态 (Driver State)

快照必须携带当前行动权驱动器的完整最小状态，但不同驱动器以**互斥分支**形式存在：

- **若使用 AV 时间轴：**
    
    - `base_distance`
    - `entity_runtime_av_map`
    - 时间轴内部优先队列或其可重建等价表示
    - 当前冻结/插队相关运行时状态
    
- **若使用经典严格回合制：**
    
    - `action_queue`
    - `cursor`
    - `round_counter`
    - 当前轮次阶段性上下文

这与[[timeline system|文档 1：时间轴与行动值系统的底层数学模型设计]]和[[classical turn driver|补充文档 1-C：标准严格回合制组件设计规范]]保持一致。

## 3.3 实体记录 (Entity Records)

`entities` 字段中，每个实体应被表示为纯数据记录，而非对象树：

- `entity_id`
- `archetype_id` / `template_id`
- `faction_id`
- `liveness_state`
- `position_payload`
- `base_attributes`
- `resource_payload`
- `runtime_tag_state`
- `buff_instance_ids`
- `custom_payload`

其中：

- `base_attributes` 用于描述实体基础面板与持久基础数值。
- `resource_payload` 用于描述当前生命、能量、护盾等资源态。
- `runtime_tag_state` 用于表达需要跨网同步的瞬时状态标签。
- `custom_payload` 用于容纳具体游戏规则定义的额外持久状态。

## 3.4 Buff 实例记录 (Buff Instance Records)

Buff 不应被序列化为监听器对象，而应序列化为可重建的**实例状态**：

- `buff_instance_id`
- `buff_type_id`
- `owner_id`
- `stack_count`
- `duration_anchor`
- `remaining_duration`
- `internal_state_payload`
- `modifier_payload`

客户端收到后，应通过本地的 Buff 工厂或注册表重新实例化 Buff，并将其监听器与修饰器重新挂载到世界中。

这与[[attributes and buffs|文档 2：实体、属性管线与 Buff 监听器设计规范]]保持一致，并可避免“幽灵监听器”残留。

## 3.5 绝对禁止同步的内容

以下内容不允许直接出现在快照中：

- Python 对象引用
- 监听器函数闭包
- UI 组件状态
- 渲染层缓存
- 已展开的工厂实例
- 任何依赖本地内存地址才能成立的数据

## 4. 覆盖哲学：ID 寻址优先于对象引用

本引擎在快照覆盖场景中，采用**ID 寻址 + 世界重建**作为标准路线，而非“对象引用保持 + 就地覆写”作为默认方案。

核心原因如下：

- **指针无法穿过网线。**
    
    网络协议、事件总线与快照天然都以 ID 交流；若底层另维持一套代理指针系统，将产生额外映射与同步风险。
    
- **ID 寻址允许世界被暴力销毁并彻底重建。**
    
    客户端可直接清空本地实体字典，再按快照重新孵化对象，而不必对旧对象逐个缝合。
    
- **表现层不持有实体对象，避免幽灵对象。**
    
    UI、动画与特效层应只缓存 `Entity_ID`，每帧向世界查询当前实体数据，而不是长期持有对象引用。

因此，本项目在状态同步层面固化如下哲学：

**万物皆 ID，世界即字典。**

## 5. 客户端标准覆盖流程

当客户端收到服务端权威快照时，标准流程如下：

1. **进入同步锁定态**
    
    暂停新的视觉外放管线，阻止动画、音效、飘字继续基于脏世界扩散。
    
2. **校验快照有效性**
    
    对比 `world_revision` 与 `authoritative_tick`，拒收过期快照。
    
3. **保留待重播的本地指令缓冲**
    
    不立即丢弃本地未来输入，而是先根据 `last_accepted_command_id` 标记哪些指令已被服务端吸收、哪些仍需重放。
    
4. **销毁旧世界**
    
    直接清空当前 `World` 内的实体字典、Buff 实例表、临时监听器注册表、事件总线残留上下文、驱动器运行时缓存。
    
5. **按快照重建新世界**
    
    使用本地工厂根据 `entities`、`buff_instances`、`driver_state` 与 `global_payload` 重新实例化所有运行时对象。
    
6. **重新挂载标准组件与本地服务**
    
    包括但不限于：
    
    - Timeline 或 Strict Turn 驱动器
    - 标准校验组件
    - 拦截器组件
    - 本地表现层桥接器
    
7. **触发 `OnSyncError(old_state, new_snapshot)`**
    
    将表现平滑处理权交给使用者的 UI 层。
    
8. **若预测模式开启，则执行静默快进**
    
    将未被服务端确认的本地指令重新在新世界中以“无头模式”重放。
    
9. **解除同步锁定态**
    
    恢复正常渲染与交互。

## 6. 静默快进：覆盖之后还要追上“现在”

真正的回滚并不止于“回到真理快照”，还必须解决覆盖后的时间缝合问题。

客户端应维护一个本地 `pending_command_buffer`，其中至少记录：

- `command_id`
- `issued_local_tick`
- `serialized_command_payload`

当覆盖发生后：

1. 丢弃所有 `command_id <= last_accepted_command_id` 的本地指令。
2. 从 `authoritative_tick` 对应的快照世界出发。
3. 按原始顺序重放剩余本地指令。
4. 将世界静默推演回当前本地时刻。

## 6.1 无头模式 (Headless Mode)

为了让“静默快进”真正可用，`Core` 必须支持无头模式：

- 允许逻辑结算。
- 禁止动画、音效、飘字、镜头抖动等视觉副作用。
- 禁止再次把同一批重播指令重新发往网络。
- 禁止重复弹出本地校验失败提示。

> **[推断补完说明]**
> 
> 历史讨论明确提出了“静默重播 96~100 帧输入”的需求，但未进一步拆解无头模式接口。本子文档将其固化为 `Core` 的必要能力，而不是可有可无的表现层技巧。

## 7. 全量与增量：先保证保底，再谈压缩

本子文档讨论的是**全量状态快照**，它是引擎在预测失败时的保底方案。

因此，当前推荐策略为：

- **基线能力：**
    
    服务端必须能随时生成一份完整但“仅含最小重建原料”的全量快照。
    
- **可选优化：**
    
    在未来另行讨论增量快照、字段压缩、字典编码、分块同步等高级策略。

换言之：

- 全量快照负责“绝对正确”。
- 增量快照负责“更省带宽”。

前者是必需品，后者是优化项。

## 8. 安全红线

若要让 3-C 真正可落地，后续实现必须遵守以下红线：

- 表现层不得长期缓存实体对象引用，最多缓存稳定 ID。
- 快照只同步数据，不同步可执行逻辑。
- Buff 与监听器必须可由本地注册表重建，不能依赖旧对象残留。
- 世界覆盖时，必须清空旧总线的残余上下文栈与临时流。
- 无头模式必须是 `Core` 的正式能力，而不是 UI 层的“静音技巧”。


# 📄 主文档 3-B：事件总线流转管线与上下文拍扁算法

**(Event Bus Pipeline & Context Flattening Algorithm)**

## 1. 核心定位与流转矛盾

中央事件总线（Event Bus）是引擎引擎底座（Core）的心脏，负责接收、调度和下发所有事件。

在内存中，由技能触发的被动、反击等连锁反应呈现为复杂的**树状结构（Event Tree）**；但网络协议要求单次 Tick 产生的所有事件必须序列化为**一维数组（Flat Array）**。总线必须在不增加业务开发者心智负担的前提下，完成树的“隐形编织”与“拍扁”。

## 2. 核心解法：动态上下文栈与自动绑定

为解放开发者（无需手动指定 `parent_id`），总线内置了 `_active_event_stack`（活跃事件栈）以实现隐形上下文追踪：

1. **宣告上下文：** 当总线开始结算事件 A（调用其 `OnExecute`）前，将 A 压入栈顶。
    
2. **静默绑定：** 在 A 结算期间，若触发了新事件 B，总线在接收 B 时自动窥探栈顶，并将 B 的 `parent_event_id` 设为 A。
    
3. **退栈与拍扁：** A 结算完毕后出栈。同时，所有**真正完成结算**的事件会被按序追加到 `_flush_stream` 数组中。
    
4. **网络下发：** 单次主循环结束时，将 `_flush_stream` 转化为 JSON 发送。客户端利用 `parent_event_id` 以 $O(N)$ 复杂度即可瞬间重构因果树。
    

## 3. 基础执行策略：DFS 深度优先嵌套

在没有任何外部组件干预的情况下，Core 默认采用 **深度优先 (DFS)** 逻辑处理衍生事件。

当 A 触发 B 时，总线会立即挂起 A 的后续结算，优先将 B 压栈并执行。B 结算完毕后，A 再继续。这完美契合绝大多数 RPG“即时触发、即时结算”的直觉物理法则。

## 4. 终极解耦：拦截器管道 (Interceptor Pipeline)

为坚守微内核架构，Core 的 Event Bus 绝不硬编码任何复杂的时序逻辑（如延迟结算、后发先至）。

总线在入口处暴露了 `IEventInterceptor` 接口。

当任何事件被 `push` 进总线，尚未进入排队与校验管线前，均会先经过所有已注册拦截器的过滤。这为上层组件截获事件、改变时序提供了绝对自由的切入点。例如[[LIFO Chain via Interceptor]]

# 📄 子文档 3-B-2：开发者最佳实践——事件批处理与同时性

**(Developer Best Practices: Event Batching & Simultaneity)**

## 1. AOE 割裂陷阱 (The Cleave Trap)

在使用 DFS 深度优先结算时，严禁将一个群攻行为（AOE）拆分为多个独立的单体事件推送给总线（例如：不应连续 push `DamageEvent_A` 和 `DamageEvent_B`）。

**危险推演：** 若分别推送，总线结算 `Damage_A` 时可能触发 A 的“全体回血”被动，该回血事件会插队执行，导致本应在爆炸中阵亡的 B 被违背常理地救活。

## 2. 核心原则：物理同时性约束

- 任何在游戏世界中应当“绝对同时”发生的改变，**必须被封装在单一的复合事件（Batch Event）中**。
    
- **正确做法：** 生成单一的 `DamageEvent(target_ids=[A, B, C])`。
    
- **引擎处理：** 总线在执行该事件的 `OnExecute` 时，在同一帧内原子化（Atomically）扣除所有目标的血量。结算完毕后，再统一收集所有存活或死亡目标触发的被动衍生事件。这就从根本上杜绝了时序割裂。

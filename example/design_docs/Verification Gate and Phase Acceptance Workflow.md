# 验证门与阶段验收流程

## 1. 文档定位

本文件定义一个**跨阶段长期有效**的标准流程：

- 每个执行阶段在宣布“可验收”前，都应先过一次统一验证门。
- 这套验证门不是某个阶段的独立功能主线，而是所有阶段共享的验收前流程。

本文件服务两个目标：

1. 防止阶段收口时只看“功能做了没有”，忽略文档、帮助、配置、测试与手测同步。
2. 防止“确定性与验证深化”在阶段规划里反复漂移成第二条并列主线。

---

## 2. 适用范围

本流程适用于所有后续执行阶段，包括但不限于：

- 调度系统深化
- 同步与恢复深化
- transport / host 加固
- 新 driver 接入
- CLI / 调试工具增强

若某阶段最终无需手测，应在阶段文档中明确说明原因，而不是直接跳过本流程。

---

## 3. 标准流程

### 3.1 阶段与范围复核

在准备收口前，先确认：

- 当前阶段命名是否准确
- 本轮实现是否仍落在已声明范围内
- 是否把 backlog 内容提前写成当前阶段任务

这一环节优先由：

- `stage-reviewer`

负责只读审查。

### 3.2 自动化验证

默认至少应完成：

- 本阶段新增或修改区域的针对性测试
- 全量 `pytest`
- `compileall`

若某阶段只改文档或 agent 配置，可明确说明“本轮未改代码，因此未重跑自动化”。

### 3.3 文档与帮助同步

若本轮改动涉及下列任一项，必须同步：

- 阶段范围文档
- 手动测试指南
- `Project Master Checklist.md`
- `Global Phase Map and Current Position.md`
- `AGENTS.md`
- `.codex/agents/README.md`
- CLI 帮助与指令参考

若本轮新增或修改的是**作者化入口**，还必须同步：

- 至少一份放在 `design docs/tooling/` 的面向使用者 usage guide
- 该 usage guide 应符合 `design docs/tooling/Authoring Documentation Standard.md`
- 对应阶段文档中的“如何使用 / 如何观察”手测入口

若新增或修改 CLI 指令，必须同时维护：

- `demo/command_reference.py`
- `design docs/CLI Command Reference.md`
- 对应回归测试

### 3.4 手动验证

以下情况默认需要手测：

- CLI 行为变化
- transport / host 行为变化
- predictive / replay / resync 行为变化
- 新 driver 切片
- 新调试注入与恢复路径

手测入口应固定成单独文档，避免只留在对话里。

### 3.5 非对称注入验证

若本轮改动涉及下列任一类状态：

- driver 内部调度状态
- predictive / replay / recover
- snapshot 覆盖
- transport 断链后恢复

则默认还应补一轮**非对称注入验证**，也就是刻意制造 client/server 或 authority/projection 的 world 状态不一致，再验证：

- 权威覆盖是否生效
- projection 是否回到正确状态
- pending / token / actor / timeline payload 是否重新对齐

该类验证优先复用：

- `tests/support/scenario_tools.py`

而不是每次临时手写 patch。

### 3.6 状态板回写

在宣布阶段完成前，应回写：

- 当前阶段是否已验收
- 采用了哪些自动化验证
- 是否完成手测
- 下一次重新回到哪个“规划门”位置

### 3.7 下一阶段说明

当阶段在本轮对话中被正式收口时，收口说明不应只给出一个“下一阶段标题”。

至少应说明：

- 下一阶段的主目标
- 为什么当前阶段到这里可以结束
- 下一阶段明确做什么
- 下一阶段明确不做什么
- 用户应依据哪些取舍来判断当前阶段是否该在这里收口

若用户本轮明确要求“工作直到某阶段”，则这一项视为收口必需内容。

### 3.8 大阶段收口复核

若当前窄阶段实际上属于某个更大的连续建设阶段，例如：

- 一整轮 `timeline T1`
- 一整轮 `C/S` 同步、恢复、诊断与验证基线建设

这里的“大阶段”仅用于**规划复核与收口判断**。

它不应用来：

- 改写已经落地的程序结构分层
- 新增一层文档存储目录层级
- 替代现有按执行阶段归档的文档组织方式

则在窄阶段收口时，还应额外复核：

- 当前大阶段是否已经接近收口点
- 本轮是否已经足以把该大阶段从“主建设中”转为“长期维护”
- 下一窄阶段是继续补同一大阶段，还是应该切换到新的主问题轴

这一步不能只留在口头说明里，至少应同步到：

- `Global Phase Map and Current Position.md`
- 对应的 planning-gate 候选文档

若判断“下一窄阶段很可能是该大阶段的收口点”，则应在候选文档中显式写出，便于用户在阶段验收时据此判断是否该停。

---

## 4. 验收输出最小集合

每个阶段收口时，最少应留下：

1. 阶段范围文档
2. 手测指南
3. 自动化验收测试或等价测试覆盖
4. 状态板更新
5. 若有 CLI 变化，对应帮助与参考文档同步
6. 若本轮接近某个大阶段收口点，对应的大阶段复核说明

---

## 5. Reviewer 分工

本流程的默认只读审查位为：

- `stage-reviewer`
  - 负责阶段命名、范围与越界检查
- `verification-reviewer`
  - 负责验收前流程检查

其中 `verification-reviewer` 重点检查：

- 自动化验证是否完成
- 手测是否有正式入口
- 文档/帮助/配置是否同步
- 阶段收口时是否遗漏状态板、验收测试或交接入口

---

## 6. 当前结论

从本文件生效开始：

- “确定性与验证深化”不再默认作为下一执行阶段的并列主线。
- 它首先作为**所有阶段共享的验收前标准流程**存在。
- 若未来真的单独启动“验证强化阶段”，也应是在本流程之外、另有独立目标与边界的阶段。

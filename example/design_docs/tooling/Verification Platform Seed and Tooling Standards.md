# 验证平台雏形与工具链标准

## 1. 文档定位

本文件用于定义一个**跨阶段长期存在**的工具链方向：

- 引擎验证平台雏形（Verification Platform Seed）
- 未来同类开发工具的组织标准

它不是某个单独阶段的范围文档，因此不归档到 `design docs/stages/`。

---

## 2. 当前判断

项目当前已经不只是“有一些测试”。

它已经逐步形成了一个验证平台雏形，主要体现在：

- 双 driver 的状态与恢复验证
- 多 transport 的远端行为验证
- predictive / replay / resync / recover 的组合验证
- CLI 诊断与调试注入
- 非对称注入与 authority 覆盖检查
- 统一验证门与阶段验收流程

这说明后续的验证系统不应再被视为零散测试集合，而应被视为：

**一个会长期随项目共同演进的开发工具链。**

---

## 3. 与阶段文档的关系

某个阶段可以承担验证平台的一小段建设任务，例如：

- 固定一组 cross-driver matrix 场景
- 补一类 fault injection
- 加一条新的恢复诊断视图

但阶段文档只负责回答：

- 本轮做了什么
- 本轮不做什么
- 这一小段如何验收

而验证平台本身的长期身份、结构标准与演化规则，应记录在本目录中。

---

## 4. 推荐工程形态

验证平台后续应优先演化为：

**tool + library**

而不是一开始就直接等同于：

- skill
- MCP
- 独立应用
- 独立仓库

推荐优先顺序：

1. repo 内可运行的验证核心
2. repo 内 CLI / runner / scenario 工具
3. 结构化报告与摘要接口
4. 需要时再暴露 MCP
5. 需要时再给 agent 增加 skill 封装
6. 只有协议和节奏足够稳定时，才考虑独立应用或独立代码库

---

## 5. 推荐分层

若后续将其正式工程化，建议按以下层次组织：

- `core`
  - 验证合同、状态摘要、统一断言
- `scenarios`
  - 固定语义场景，而不是只按 CLI 命令命名
- `runners`
  - local / subprocess / socket 等执行器
- `faults`
  - 非对称注入与故障注入
- `reports`
  - JSON / 摘要 / diff / phase gate 输出
- `cli`
  - 面向开发者的直接入口
- `adapters`
  - MCP、skill 等外部适配层

---

## 6. 复杂度控制原则

验证平台必须显式避免无脑笛卡尔积。

后续默认采用：

- 分层合同测试
- 稀疏 determinism matrix
- 阶段增量扩展

而不是：

- 每加一个 driver 就把所有场景乘一遍
- 每加一个 transport 就把所有 recovery 全复制一遍
- 每个新能力都直接升格为 matrix 维度

核心原则：

1. 新增组件默认不新增 matrix 维度
2. 新增能力先落局部合同测试，再决定是否进入 matrix
3. matrix 按能力标签选代表组合，不做全笛卡尔积
4. 每条主线只保留少量 canonical 场景
5. 每个阶段只补最小必要覆盖

---

## 7. 与 skill / MCP / 独立工具的关系

后续若要把验证平台暴露给 agent 或其他系统，应遵循：

- skill：只作为 AI 工作流说明层
- MCP：只作为结构化访问适配层
- CLI/tooling：仍是事实来源

也就是说：

- skill 不是验证平台本体
- MCP 不是验证平台本体
- 真正的源事实应始终来自项目内可运行的验证工具链

---

## 8. 当前结论

从当前节点开始：

- 验证系统应被视为一个长期发展的平台雏形
- 其阶段性建设可以继续通过 `stages/` 中的单一窄主线推进
- 但其长期定义、结构标准与工具链身份，应统一维护在 `design docs/tooling/`

# 问题反馈 004: 同版本重打包会导致默认安装路径静默跳过更新

## 严重性

- Medium

## 问题类型

- 发布管理 / 安装路径一致性

## 现象

- `doc_loop_vibe_coding-0.9.3-py3-none-any.whl` 在 2026-04-19 有新的文件时间戳和内容变化，但内部版本号仍为 `0.9.3`。
- 当前环境中已安装 `doc-loop-vibe-coding 0.9.3` 时，普通安装路径不会自动替换为新的同版本内容。
- 本轮测试必须使用 `pip install --force-reinstall --no-cache-dir` 才能确保拿到更新后的 wheel 内容。

## 复现步骤

1. 在环境中先安装旧的 `doc-loop-vibe-coding 0.9.3`。
2. 用新的同名同版本 wheel 替换 workspace 中的文件。
3. 执行普通安装而不是 `--force-reinstall`。
4. 观察 pip 不会把同版本包视为需要更新。

## 预期结果

- 新内容发布时应具备可识别的版本变化，或提供明确的安装指引，避免用户以为“已经更新”但实际仍在运行旧内容。

## 实际结果

- 同版本重打包需要使用强制重装参数，默认安装路径存在静默跳过更新的风险。

## 影响

- 测试者和用户可能认为自己已经装上新包，实际仍在运行旧内容。
- 自动化安装脚本如果未显式加 `--force-reinstall`，会得到不确定结果。

## 建议修复

- 对内容发生变化的发行包执行版本递增。
- 若必须保留同版本重打包，需在安装文档中明确要求 `--force-reinstall` / `--force`。
- 在扩展诊断输出或安装日志中提示“当前版本相同，但可能需要强制更新”。
